# Founders Inbody 확장 설계안

이 문서는 다음 3가지를 한 번에 정리한 실행용 설계안이다.

1. 업종 체계 재설계
2. 업종별 추가 질문 설계
3. 결과 저장 / 다시보기 구조 설계

목표는 "범용 진단"에서 끝나지 않고, 업종과 숫자 흐름에 맞는 더 뾰족한 결과를 만들 수 있는 기반을 만드는 것이다.

---

## 1. 업종 체계 재설계

### 1-1. 현재 구조의 한계

현재 업종 분류는 MVP로는 충분하지만, 실제 액션 제안까지 맞춤화하려면 몇 가지 한계가 있다.

- `SERVICE`가 너무 넓어서 결과가 평평해진다.
- `교육·컨설팅`이 빠져 있어 대표적인 B2B/지식서비스 회사가 적절히 분류되지 않는다.
- `마케팅`과 `IT`는 들어가 있지만, 매출 구조가 크게 다른 `콘텐츠`, `이커머스`, `플랫폼`, `오프라인 매장형`이 빠져 있다.
- 같은 업종 안에서도 매출 공식이 완전히 다를 수 있는데, 업종 분류가 이를 충분히 반영하지 못한다.

### 1-2. 추천 업종 체계

추천은 "업종명"이 아니라 "경영구조가 다른 사업군" 기준으로 재분류하는 것이다.

#### A. 제조 / 유통 / 상품형

- 제조업 B2B
- 제조업 B2C
- 유통·도소매
- 이커머스·D2C
- 헬스케어 제품

#### B. 서비스 / 인력가동형

- 교육·강의
- 컨설팅·코칭
- 에이전시·대행업
- 전문서비스
  - 법무
  - 회계·세무
  - 노무·인사
  - 디자인·설계
- 헬스케어 서비스
- 뷰티·라이프스타일 서비스

#### C. 기술 / 플랫폼형

- IT·SaaS
- 플랫폼·중개
- 헬스케어 테크
- 콘텐츠·미디어

#### D. 오프라인 자산 / 공간형

- 의료 (일반 진료)
- 의료 (비급여·시술)
- 부동산 중개·임대
- 부동산 개발·시행
- 오프라인 매장·프랜차이즈

### 1-3. 앱에 우선 추가하면 좋은 업종

1차 우선순위:

- 교육·강의
- 컨설팅·코칭
- 이커머스·D2C
- 콘텐츠·미디어
- 플랫폼·중개
- 오프라인 매장·프랜차이즈
- 전문서비스

이 7개만 추가해도 결과 품질이 눈에 띄게 올라간다.

### 1-4. 업종 추가 원칙

업종을 늘릴 때는 이름이 아니라 아래 기준을 만족할 때만 새 업종으로 분리한다.

- 매출 공식이 다르다.
- 핵심 KPI가 다르다.
- 대표가 의사결정할 때 보는 레버가 다르다.
- 액션 카드가 달라진다.

예를 들어 `교육·강의`와 `컨설팅·코칭`은 둘 다 지식서비스처럼 보이지만 실제로는 다르다.

- 교육: 수강생 수, 재수강률, 객단가, 기업교육 비중
- 컨설팅: 프로젝트당 매출, 리테이너 비중, 대표 의존도, 수주 파이프라인

따라서 분리하는 것이 맞다.

## 2. 업종별 추가 질문 설계

### 2-1. 기본 방향

지금 앱은 공통 질문 + 매출 공식 질문으로 설계되어 있다. 이 구조는 유지하되, 업종별로 2~4개의 "뾰족 질문"을 추가하는 방식이 가장 좋다.

추천 흐름:

1. 공통 질문
2. 분기 / 연간 흐름 질문
3. 매출 공식 질문
4. 업종별 추가 질문 2~4개
5. 결과 생성

이렇게 하면 너무 길어지지 않으면서도 업종 특화 진단이 가능하다.

### 2-2. 업종별 추천 질문

#### 교육·강의

핵심 지표:

- 재수강률
- 강의당 평균 객단가
- B2B 매출 비중
- 대표 직접 강의 의존도

추가 질문 예시:

- 최근 12개월 수강생 수는 몇 명입니까?
- 그중 재수강한 고객은 몇 명입니까?
- 기업교육 매출 비중은 몇 %입니까?
- 대표가 직접 진행한 매출 비중은 몇 %입니까?

진단 포인트:

- 재수강이 낮으면 콘텐츠 품질/후속 상품 구조 이슈
- 대표 의존도가 높으면 확장성 제약
- B2B 비중이 높으면 리드타임 긴 대신 안정성 높음

#### 컨설팅·코칭

핵심 지표:

- 리테이너 비중
- 프로젝트당 평균 매출
- 대표 직접 수행 비중
- 파이프라인 확보 개월 수

추가 질문 예시:

- 최근 12개월 매출 중 월정액·리테이너 비중은 몇 %입니까?
- 평균 프로젝트 단가는 얼마입니까?
- 대표가 직접 수행해야만 하는 매출 비중은 몇 %입니까?
- 현재 확보된 수주 파이프라인은 몇 개월치입니까?

진단 포인트:

- 리테이너 비중이 낮으면 매출 변동성 큼
- 대표 직접 수행 비중이 높으면 조직화가 안 된 상태
- 파이프라인 부족이면 분기 흔들림이 큼

#### 이커머스·D2C

핵심 지표:

- 재구매율
- CAC 또는 광고비 비중
- 반품률
- 공헌이익

추가 질문 예시:

- 최근 3개월 재구매 고객 비중은 몇 %입니까?
- 최근 분기 광고비는 얼마입니까?
- 반품률은 몇 %입니까?
- 대표 상품 상위 3개가 매출의 몇 %를 차지합니까?

진단 포인트:

- 매출이 커도 반품률/광고비가 높으면 실제 이익 약함
- 상품 집중도가 높으면 리스크 큼

#### 콘텐츠·미디어

핵심 지표:

- 유료 전환율
- 광고/협찬 매출 의존도
- 채널 집중도
- 대표·특정 크리에이터 의존도

추가 질문 예시:

- 최근 12개월 매출 중 광고·협찬 비중은 몇 %입니까?
- 유료 구독 또는 유료 상품 전환율은 몇 %입니까?
- 상위 채널 1개가 전체 매출의 몇 %를 만듭니까?
- 대표 개인 브랜딩 의존도는 몇 %입니까?

진단 포인트:

- 특정 채널 의존이 높으면 외부 플랫폼 리스크 큼
- 광고 의존이 높으면 경기 변동 민감

#### 플랫폼·중개

핵심 지표:

- 활성 공급자 수
- 활성 수요자 수
- 거래 전환율
- take rate
- 리텐션

추가 질문 예시:

- 최근 분기 활성 판매자 수는 몇 개입니까?
- 최근 분기 활성 구매자 수는 몇 명입니까?
- 거래 성사율은 몇 %입니까?
- 총 거래액 대비 수수료율은 몇 %입니까?

진단 포인트:

- 거래액보다 take rate와 리텐션이 중요
- 양면시장 균형이 핵심

#### 오프라인 매장·프랜차이즈

핵심 지표:

- 점포당 매출
- 객수
- 객단가
- 인건비 비중
- 좌석/회전율

추가 질문 예시:

- 최근 분기 평균 객수는 몇 명입니까?
- 평균 객단가는 얼마입니까?
- 인건비가 매출의 몇 %입니까?
- 매장 또는 좌석 회전율은 어느 정도입니까?

진단 포인트:

- 객수/객단가/회전율 중 어디가 병목인지 구분 가능

#### 전문서비스

핵심 지표:

- 청구 가능 시간 비율
- 리테이너 비중
- 고객 집중도
- 인력 가동률

추가 질문 예시:

- 최근 12개월 매출 중 상위 3개 고객 비중은 몇 %입니까?
- 리테이너 매출 비중은 몇 %입니까?
- 인력의 청구 가능 시간 비율은 몇 %입니까?
- 대표 검토 없이 처리 가능한 업무 비중은 몇 %입니까?

진단 포인트:

- 상위 고객 집중도가 높으면 한 고객 이탈 리스크 큼
- 대표 승인 병목이 구조 문제로 연결됨

### 2-3. 질문 설계 원칙

업종별 질문은 다음 원칙으로 설계한다.

- 대표가 모를 가능성이 너무 높은 질문은 피한다.
- 계산 가능한 숫자를 우선 받는다.
- 민감하지만 중요한 항목은 범위 선택으로 받을 수 있게 한다.
- 질문은 2~4개를 넘기지 않는다.

### 2-4. 구현 우선순위

1차 구현:

- 교육·강의
- 컨설팅·코칭
- 이커머스·D2C
- 콘텐츠·미디어

2차 구현:

- 플랫폼·중개
- 오프라인 매장·프랜차이즈
- 전문서비스

## 3. 결과 저장 / 다시보기 구조 설계

### 3-1. 요구사항 정리

사용자가 원하는 것은 사실상 아래 4가지다.

- 결과를 다시 볼 수 있어야 함
- API를 쓰든 안 쓰든 결과가 유지돼야 함
- 로그인 없어도 쓸 수 있으면 좋음
- 나중에 PDF나 이미지 다운로드와도 연결 가능해야 함

### 3-2. 결론: IP 기준은 비추천

IP 기준 저장은 가능하지만 추천하지 않는다.

이유:

- 회사/카페/공용망에서는 여러 명이 같은 IP를 쓴다.
- 모바일 환경에서는 IP가 자주 바뀐다.
- 사용자가 집/회사/휴대폰을 바꾸면 다시 못 찾을 수 있다.

추천 방식은 `resultId` 기반 저장이다.

### 3-3. 추천 저장 방식

가장 추천:

- DB에 결과 저장
- 저장 시 `resultId` 발급
- 결과 확인 URL: `/result/[resultId]`

보조 저장:

- 브라우저 `localStorage`에 최근 결과 ID 목록 저장
- 사용자가 다시 방문하면 "최근 결과 다시보기" 표시

선택 옵션:

- IP + userAgent + createdAt 을 보조 메타데이터로 저장
- 단, 조회 키로는 사용하지 않음

### 3-4. 저장해야 하는 데이터 구조

추천 스키마:

```ts
type ResultRecord = {
  id: string;
  createdAt: string;
  updatedAt: string;
  version: string;
  industry: string;
  input: {
    // 모든 원본 입력값
  };
  derived: {
    // 계산된 경영지표
  };
  diagnosis: {
    type: string;
    headline: {
      asIs: string;
      toBe: string;
    };
    dashboard: Array<{
      label: string;
      value: string;
      note?: string;
    }>;
    actions: Array<{
      title: string;
      fact: string;
      insight: string;
      action: string;
      checklist: string[];
    }>;
    metricGuide: Array<{
      title: string;
      body: string;
    }>;
  };
  aiResult?: {
    model: string;
    promptVersion: string;
    responseJson: unknown;
  };
  meta?: {
    ipHash?: string;
    userAgent?: string;
  };
};
```

핵심은 `input`, `derived`, `diagnosis`, `aiResult`를 분리하는 것이다.

- `input`: 사용자가 넣은 사실
- `derived`: 코드로 계산한 숫자
- `diagnosis`: 화면에 렌더링하는 결과
- `aiResult`: API 붙인 경우의 추가 해석

### 3-5. API를 빼더라도 다시보기 가능한 구조

가능하다. 그리고 그렇게 설계하는 것이 맞다.

원칙:

- 화면에 보여준 최종 결과 JSON을 저장한다.
- 다시보기는 "재생성"이 아니라 "저장된 결과 렌더"를 기본으로 한다.

이렇게 하면:

- API 모델이 바뀌어도 예전 결과가 안 바뀐다.
- API를 아예 비활성화해도 과거 결과는 그대로 볼 수 있다.
- 버전별 비교도 가능하다.

### 3-6. 구현 단계

#### 1단계: 로컬 저장

빠르게 구현 가능:

- 결과 생성 시 `localStorage`에 최근 5개 저장
- "최근 결과 다시보기" 목록 노출

장점:

- 가장 빠름

한계:

- 브라우저/기기 바뀌면 못 봄

#### 2단계: DB 저장 + resultId

추천 MVP:

- Vercel Postgres 또는 Supabase 사용
- 결과 생성 시 DB 저장
- `/result/[id]` 페이지 생성

장점:

- 로그인 없어도 URL만 있으면 다시보기 가능
- PDF / 이미지 다운로드와 연결 쉬움

#### 3단계: 히스토리 목록

- 동일 브라우저에서는 최근 결과 목록 표시
- 필요시 메일 없이 "내 최근 진단" 화면 제공

### 3-7. API 포함 시 저장 흐름

#### API 없는 경우

1. 입력 완료
2. 코드가 지표 계산
3. 규칙 기반 결과 생성
4. 결과 JSON 저장
5. resultId 반환

#### API 있는 경우

1. 입력 완료
2. 코드가 지표 계산
3. 계산 결과를 바탕으로 기본 결과 JSON 생성
4. 서버에서 LLM 호출
5. AI 결과를 추가 병합
6. 최종 결과 JSON 저장
7. resultId 반환

핵심은 API 유무와 상관없이 `최종 결과 JSON 저장`이 동일하다는 점이다.

## 4. 추천 구현 순서

### Phase 1

- 업종 체계 개편
- 교육·강의 / 컨설팅·코칭 / 이커머스 / 콘텐츠 업종 추가
- 업종별 추가 질문 분기

### Phase 2

- 결과 저장 테이블 생성
- resultId 기반 다시보기 페이지
- localStorage에 최근 결과 ID 저장

### Phase 3

- PDF 다운로드
- 이미지 다운로드

### Phase 4

- AI 해석 API 연결
- 업종별 액션 문구 고도화
- 저장된 AI 결과의 버전 관리

## 5. 바로 구현할 때 필요한 작업 목록

### 업종

- `Industry` 타입 확장
- 업종 라벨/힌트 추가
- 업종별 benchmark 매핑 재정리
- 업종별 follow-up 질문 스키마 추가

### 질문

- `StepView`에서 업종 분기 질문 삽입
- 업종별로 2~4개씩만 추가
- 결과 생성 함수에서 업종 질문 반영

### 저장

- DB 테이블 `results` 생성
- POST `/api/results`
- GET `/api/results/[id]`
- `/result/[id]` 페이지 생성
- 저장 완료 후 resultId 기반 URL 복사 버튼 제공

### 다시보기

- localStorage에 최근 resultId 저장
- 홈에 "최근 결과 다시보기" 섹션 표시

## 6. 권장 의사결정

가장 추천하는 다음 구현 순서는 다음과 같다.

1. 업종을 먼저 정리한다.
2. 업종별 질문을 1차 4개 업종에만 붙인다.
3. 결과 저장과 다시보기를 `resultId` 방식으로 붙인다.
4. 그 다음에 API를 붙여 액션 문구를 업종/숫자 기반으로 풍성하게 만든다.

IP 기준 저장은 보조 메타데이터로만 쓰고, 식별 기준으로는 쓰지 않는 것을 권장한다.
