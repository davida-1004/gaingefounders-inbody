# Founders Inbody v3 — page.tsx 사용 가이드

> 단일 파일(`page.tsx`) 하나로 Next.js 프로젝트에 그대로 붙이면 돌아갑니다. TypeScript 타입 체크 0건 에러.

---

## 1. 배포 방법

### 이미 Next.js 프로젝트가 있는 경우

프로젝트의 `app/page.tsx` 파일을 이번에 만든 `page.tsx` 로 **그대로 덮어쓰기** 하시면 됩니다.

외부 의존성은 **React + Tailwind CSS** 뿐입니다. 다른 패키지 설치 필요 없습니다.

### 처음 프로젝트를 만드시는 경우

터미널에서 이렇게 실행하시면 됩니다.

```bash
# 1. Next.js 프로젝트 생성
npx create-next-app@latest founders-inbody --typescript --tailwind --app
cd founders-inbody

# 2. app/page.tsx 파일을 이번에 받은 page.tsx 로 덮어쓰기
# (파일 관리자에서 복사·붙여넣기 하시면 됩니다)

# 3. 로컬에서 확인
npm run dev

# 4. Vercel에 배포
npx vercel
```

`npm run dev` 실행 후 브라우저에서 `http://localhost:3000` 을 열면 진단지가 보입니다.

## 2. 이 파일이 하는 일

### 질문 흐름 (최대 10~11문항)

| # | 질문 | 입력 |
|---|---|---|
| 1 | 업종 | 15개 중 선택 |
| 2 | 매출 유형 | 민간/공공/병행/지원 |
| 3 | 연 매출 | 만원 |
| 4 | 영업이익 | 만원 (음수 허용) |
| 5 | 가용 현금 | 만원 |
| 6 | 월 고정비 | 만원 |
| 7 | 정규 인원 | 명 |
| 8 | 매출 공식 | A/B/C/D/E 5개 중 |
| 9~11 | 공식별 분기 질문 | A·B·E는 3문항, C·D는 2문항 |
| +추가 | 매출 유형별 분기 | MIX 1개, GOV 2개, PUBLIC 1개, PRIVATE 0개 |

### 결과 화면 6단 구조

- **01 한 줄 진단** — 메인 유형 4개 + As-is/To-be 문장 + 업종 특이점 주석
- **02 매출 공식** — 공식 표현 + 해석
- **03 이번 달 집중 지표** — GCS 단계별 자동 선정 (런웨이 / 재구매율 / 전환율 / 재계약률 / 영업이익률 중 하나)
- **04 행동 전략 3가지** — F-I-A-C 구조 액션 카드 (Data → Why → Action → Checklist)
- **05 현금 운용 우선순위** — GCS 3단계 (Survive / Cash / Grow) 자동 판정
- **06 경영 계기판** — 영업이익률·런웨이·인당매출·HCROI·(지원금 의존도)

### 내부에서 자동 계산되는 것

```
영업이익률 = 영업이익 ÷ 매출 × 100
현금 런웨이 = 현금 ÷ 월 고정비
HCROI = 영업이익 ÷ (인원 × 6000만원)
인당 매출 = 매출 ÷ 인원
지원금 의존도 = 지원금 ÷ 매출 × 100 (MIX/GOV)
민간 매출 커버력 = (매출-지원금)/12 ÷ 월 고정비 (MIX)
재구매율 = A2 ÷ A1 (A형)
전환율 = B2 ÷ B1 (B형)
재계약률 = D2 ÷ D1 (D형)
SME 여부 = 인원 < 300
GCS 단계 = Survive(런웨이<3 or 영업적자) / Cash(3~6) / Grow(6+)
```

## 3. 벤치마크 — 한국은행 2024

결과 화면의 업종 평균 수치는 전부 **한국은행 「2024년 기업경영분석」 확정판 (2025.10 공표)** 기준이며 중소기업 평균(SME)을 기본으로 노출합니다.

**내장된 벤치마크 14개**:
- 전 산업 · 제조업(전체/식료품/의약품/의료기기) · 도소매(전체/오프라인) · 숙박음식 · 정보통신(전체/소프트웨어) · 부동산 · 전문과학기술 · 광고업 · 예술스포츠여가

15개 세부 업종은 이 14개 중 가장 가까운 것으로 자동 매핑됩니다.

## 4. 내부 설계 원칙 (참고용)

### 원칙 1: 프레임워크 이름은 UI에 나타나지 않음

As-is/To-be, PQC, STP, GCS, What Tree, Why Tree, F-I-A-C, VRIO, 5 Forces, BCG 등 10개 프레임워크가 진단 엔진에 녹아있지만, **대표 화면에는 이 이름들이 한 번도 보이지 않습니다**. 대표는 자기 상황에 맞는 한국어 문장만 받습니다.

### 원칙 2: 앱은 자기가 아는 것만 말함

액션 카드의 **Why(Insight) 블록**은 엄격한 규칙을 지킵니다.
- 허용: Fact의 숫자 해석 · 일반 경영 원리 · 앱이 수행한 시뮬레이션
- 금지: 검색어, 유입 소스, 해지 고객 사유, 고객 연령 분포 등 앱이 알 수 없는 구체 사실

구체 확인이 필요한 정보는 **Action 블록의 확인 행동**("Google Analytics 에서 경로별 전환율 확인하십시오" 등) 으로 돌립니다.

### 원칙 3: 원자료만 입력, 비율은 전부 앱이 계산

대표는 영업이익률·HCROI 같은 **계산된 지표를 입력하지 않습니다**. 원자료(매출, 영업이익, 인원 등)만 입력하고 비율·배수·개월수는 전부 앱이 자동 계산합니다.

## 5. 앞으로 추가할 것 (로드맵)

이 MVP 에서는 다음이 생략되었습니다. 실 운영 후 피드백 받아서 순차 추가하시면 됩니다.

- **액션 카드 풀 확장**: 현재는 유형별 3개만. 나중에 `templates/actions.ts` 로 60개 풀 분리
- **운영 리스크 플래그**: 대표 병목·조직 계층 질문 (v3 기획서 §3-5) — 인원 3명 이상일 때만 노출
- **2차 What Tree 분기**: 메인 유형별 3축 심화 질문
- **후속 워크시트**: VRIO(자기 무기 찾기), BCG(상품 포트폴리오) 매트릭스
- **결과 저장·공유**: 진단 결과를 URL 공유 또는 PDF 내보내기

현재 상태로도 **리더·테스트 유저에게 보여주기 충분한 완성도**입니다.

## 6. 파일 구조 요약

```
page.tsx (1,338줄)
├── Part 1   타입 정의
├── Part 2   업종 메타 (15개)
├── Part 3   한국은행 2024 벤치마크 (14개)
├── Part 4   computeDerived — 파생지표 계산
├── Part 5   classify — 메인 유형 판정
├── Part 6   buildDiagnosis — 결과 조립 (헤드라인·공식·레버·액션·현금·계기판)
├── Part 7   React UI 루트
├── Part 8   질문 흐름 제어
├── Part 9   공통 UI 컴포넌트
└── Part 10  결과 화면 렌더러
```

모든 함수는 **순수 함수**입니다. LLM API 호출 없음. localStorage 사용 없음.
